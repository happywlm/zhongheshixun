## 1 开发环境搭建与主流框架整合

### 1.1 统一框架下的咨询模块搭建

平台后端以 Spring Boot 2.7.18 为基座，通过 Maven 父工程统一管理依赖（MyBatis-Plus 3.5.3、MySQL 8.0、jjwt 0.11.5、Hutool 等）。本人负责的咨询后端位于 `training-api` 模块（端口 9899），与 `training-admin`（9898）共用 `training-common`、`training-dao`、`training-service` 三层，**无需重复搭建框架**，直接复用已整合好的数据源、事务与 ORM 能力，仅编写咨询业务代码，保证框架整合一致性。

### 1.2 数据源、事务与咨询业务的一致性

咨询业务涉及"写工单 + 写关键词命中 + 更新统计"等多步写库，本人使用 Spring 声明式事务保证原子性：

```java
@Override
@Transactional(rollbackFor = Exception.class)
public AskResult ask(Long userId, String question) {
    if (!StringUtils.hasText(question)) {
        throw new IllegalArgumentException("问题不能为空");
    }
    // 1. 关键词转人工检查
    if (containsTransferHumanKeyword(question)) {
        ConsultRecord record = new ConsultRecord();
        record.setStudentId(userId);
        record.setQuestion(question);
        record.setIsAuto(0);                 // 人工工单
        consultRecordMapper.insert(record);
        return new AskResult(record.getId(), null, false, "human");
    }
    // 2. 调用 LongCat AI（未启用时返回 null，走人工兜底）
    String aiAnswer = longCatAiService.ask(question);
    if (StringUtils.hasText(aiAnswer)) {
        ConsultRecord record = new ConsultRecord();
        record.setStudentId(userId);
        record.setQuestion(question);
        record.setAnswer(aiAnswer);
        record.setIsAuto(1);                 // AI 自动回复
        record.setReplyTime(LocalDateTime.now());
        consultRecordMapper.insert(record);
        return new AskResult(record.getId(), aiAnswer, true, "ai");
    }
    // 3. 兜底：创建人工工单
    ConsultRecord record = new ConsultRecord();
    record.setStudentId(userId);
    record.setQuestion(question);
    record.setIsAuto(0);
    consultRecordMapper.insert(record);
    return new AskResult(record.getId(), null, false, "human");
}
```

`@Transactional(rollbackFor = Exception.class)` 确保任一步骤异常时整段回滚，避免出现"工单已建但关键词未记录"的脏数据。连接池沿用 HikariCP（已在父工程配置 `maximum-pool-size: 20`），支撑弱网环境下并发咨询请求。

### 1.3 对象-关系接口映射（MyBatis-Plus）

数据访问层以 MyBatis-Plus 简化映射：`ConsultRecordMapper`、`ConsultKeywordMapper`、`KnowledgeBaseMapper` 均继承 `BaseMapper<T>`，获得 `insert`、`selectById`、`selectPage` 等通用方法；需联表或定制更新的语句（如管理员回复、超时工单查询）用 XML 映射，保持可控：

```java
public interface ConsultRecordMapper extends BaseMapper<ConsultRecord> {
    int updateReply(@Param("id") Long id, @Param("answer") String answer,
                    @Param("status") int status, @Param("replyTime") LocalDateTime replyTime);
    List<ConsultRecord> selectOverdue(@Param("slaHours") int slaHours);
}
```

全局配置 `logic-delete-field: deleted` 使所有删除转为 `UPDATE ... SET deleted=1`，咨询工单不被物理删除，满足医疗数据可追溯要求。以上映射经启动 `training-api` 后调用 `/api/consult/ask` 返回工单 ID 验证整合无误。

### 1.4 微信小程序前端框架

小程序学员端采用微信原生框架（WXML/WXSS/JS），在 `app.json` 中声明页面与 tabBar：

```json
{
  "pages": ["pages/index/index", "pages/login/login", "pages/profile/profile",
            "pages/course/list/list", "pages/exam/list/list",
            "pages/plan/plan", "pages/consult/index/index", "..."],
  "tabBar": {
    "list": [
      { "pagePath": "pages/index/index", "text": "首页" },
      { "pagePath": "pages/plan/plan", "text": "计划" },
      { "pagePath": "pages/course/my-courses/my-courses", "text": "学习" },

```

网络请求统一封装 `request`，自动附加 `Authorization: Bearer <token>`，适配弱网（`timeout: 10000`）。

## 2 敏感信息脱敏/加密与 RBAC 权限控制

咨询内容包含学员真实姓名、手机号等个人敏感信息，受《个人信息保护法》《数据安全法》约束。本章说明本人负责的合规处理。

### 2.1 小程序端鉴权（wx.login）

小程序不采集账号密码，改用微信一键登录，从源头降低凭据泄露风险：

```javascript
// miniprogram/pages/login/login.js
onLogin() {
  if (!this.data.agreed) { wx.showToast({ title: '请先同意用户协议', icon: 'none' }); return; }
  wx.login({
    success: async (res) => {
      if (!res.code) return;
      const result = await userApi.login({ code: res.code, nickname: '学员', avatar: '' });
      wx.setStorageSync('token', result.token);          // 仅存 JWT
      wx.setStorageSync('userInfo', result.userInfo);
      wx.reLaunch({ url: '/pages/index/index' });
    }
  });
}
```

后端 `/api/wx/login` 用 `code` 换取 openid（演示环境以 code 模拟 openid），按 openid 查/建学员账号并签发 JWT，**全程不存储微信密码，也不暴露平台账号密码**。

### 2.2 敏感信息脱敏与加密

- **密码**：沿用统一安全基座的 BCrypt 单向哈希，库内仅存 `$2a$10$...` 形式，不可逆。
- **身份证号等强敏感字段**：按工程规范采用 AES-256 字段级加密（密钥经配置注入，不在代码硬编码），读库后仅在已授权场景解密，前端默认脱敏展示为 `5101**********1234`。
- **咨询中的姓名/手机号**：采用"存储明文 + 展示脱敏"策略，列表与工单详情以 `138****8000` 形态呈现，兼顾业务可用与隐私保护。

### 2.3 RBAC 在咨询模块的落地

平台采用 RBAC（用户—角色—权限），本人负责的咨询权限为 `consult:read`（查看工单）、`consult:write`（回复/知识库管理）。角色分配为：`ADMIN` 与 `TEACHER` 可处理咨询，`STUDENT` 仅能查看自己的咨询记录。Spring Security 中以 URL 规则控制：

```java
.antMatchers("/admin/consult/**").hasAnyRole("ADMIN","TEACHER")  // 咨询管理仅讲师/管理员
.anyRequest().authenticated()
```

小程序端学员调用 `myList(studentId)` 只能查自己 `student_id` 的工单，后端以 `wrapper.eq(ConsultRecord::getStudentId, studentId)` 强制隔离，杜绝越权查看他人咨询。

### 2.4 合规风险规避要点

编码时主动规避：① 小程序微信登录免存密码；② 身份证号 AES 加密、姓名/手机号展示脱敏，落实去标识化；③ 咨询接口最小权限，学员仅看自己；④ 生产环境传输层用 HTTPS/WSS 防窃听；⑤ 统一异常不回显 SQL 与堆栈；⑥ 咨询工单逻辑删除保留审计痕迹；⑦ 仅收集咨询必需信息，遵循最小必要原则。

## 3 工程文档体系与 Git 版本控制

### 3.1 独立文档与角色分工

为避免四人并行开发接口对不上，本人在项目中维护独立文档 `defense-materials/综合实训分工方案/04-学生丁-咨询与小程序.md`，专述丁的分工、技术点清单、代码示例与依赖关系；平台级文档（设计/开发/数据库/API）置于 `docs/`。四人分工与依赖如下：

| 学生 | 综合实训角色 | 主要任务 | 专属数据表 |
|------|--------------|----------|------------|
| 甲 | 用户体系与培训底座 | 用户/角色/培训计划 | sys_user、teacher、train_plan、plan_course |
| 乙 | 课程与学习 | 课程、章节、报名、学习进度 | course、course_chapter、course_enroll、study_record |
| 丙 | 考试与统计 | 试题、考试、组卷阅卷、统计 | exam、exam_paper、exam_record、question、knowledge_point |
| **丁** | **在线咨询与小程序** | **咨询后端 + 小程序学员端** | knowledge_base、consult_record、consult_keyword、resource_file |

文档中明确丁的依赖：小程序登录依赖甲的 `/api/wx/login`；咨询页展示课程推荐依赖乙的课程接口；咨询中引用考试成绩依赖丙的考试接口。只有"独立文档描述分工 + 契约对齐接口"，四人才能并行而不冲突。

### 3.2 Git 版本控制策略

丁的模块使用 Git 进行版本控制，采用 `main` 主干 + 特性分支的轻量模式，提交遵循 Conventional Commits：

```bash
git checkout -b feat/d-miniprogram        # 小程序学员端
git commit -m "feat(miniprogram): 微信一键登录 + 动态 baseURL 适配"
git checkout -b feat/d-consult            # 咨询模块
git commit -m "feat(consult): AI优先+关键词转人工咨询流程"
git push -u origin feat/d-consult
git checkout main && git merge --no-ff feat/d-consult
```

远程仓库已配置（`origin` 指向 GitHub 镜像）。与丁相关的真实提交示例：`feat(M14): 修复5项功能Bug + 在线咨询方案A`、`fix(M16): 修复管理后台试题/考试页面课程筛选失效`。`feat/fix/chore/docs` 前缀使意图清晰，`.gitignore` 忽略 `target/`、依赖与本地配置，保证仓库精简可追溯。

### 3.3 文档与代码一致性

文档与代码同源纳入 Git 版本管理；咨询接口或小程序页面任一变更，均同步更新 `04-学生丁-咨询与小程序.md` 并提交，避免出现"代码跑通、文档过期"的脱节。

## 4 技术选型成本分析：开源 vs 商业

作为小程序与咨询模块负责人，本人就相关技术做了开源与商业成本对比，结论为优先开源可显著降低投入（契合毕设零预算与基层机构低成本诉求）。

| 能力 | 开源方案（本项目采用） | 商业/闭源替代 | 年成本（估） | 选型结论 |
|------|------------------------|---------------|--------------|----------|
| 移动端 | 微信小程序（免费开发） | 原生 APP 外包/商业 MAM | 数万~数十万元 | 小程序免上架费 |
| 后端框架 | Spring Boot（Apache 2.0） | Oracle WebLogic / IBM WebSphere | 数万~数十万元/核·年 | 开源免费且生态成熟 |
| 持久层 | MyBatis-Plus（Apache 2.0） | 商业 ORM/代码生成工具 | 数千~数万元 | 开源国产活跃维护 |
| 数据库 | MySQL 8.0（GPL 免费） | Oracle DB / SQL Server | 数十万元起 | 开源零成本 |
| 图表可视化 | ECharts（BSD 免费） | FineReport / 商业 BI | 数万元起 | 开源满足统计需求 |
| AI 咨询 | LongCat 等开源/低价模型 | 商业智能客服系统 | 数千~数万元/年 | 开源/低价优先 |
| 版本控制 | Git + GitHub（免费） | GitLab EE / 商业托管 | 数千元/人·年 | 开源免费 |

**降本结论**：小程序免应用商店上架与外包费用，ECharts 与 MySQL/Spring Boot 零授权费，AI 咨询选用开源/低价模型替代商业客服。上述组合在功能完整、性能达标（3000 并发学习、1000 并发考试）前提下，将丁所负责模块的软件授权成本降至零；仅在需企业级支持时再考虑商业订阅。

## 5 结论

本文以学生丁的分工为主线，说明了基层卫生培训平台的在线咨询模块与微信小程序学员端工作：在开发环境上，复用 Spring Boot + MyBatis-Plus 整合好的数据源、事务与接口映射，搭建咨询后端并落地微信原生小程序；在安全上，以小程序微信登录免存密码、身份证号 AES 加密、姓名/手机号脱敏，并以 RBAC（consult:read/consult:write）+ JWT 落地权限控制，满足医疗数据合规要求；在工程化上，维护独立分工文档与 Git 版本控制，明确丁与甲、乙、丙的依赖；在选型上，经开源与商业成本对比，优先开源使模块投入降至最低。上述工作为学员提供了可用的咨询入口，并与甲、乙、丙三端无缝衔接。
