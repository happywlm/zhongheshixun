## 一、环境与基础设施问题

### 1.1 JDK 版本不匹配
- **现象**：`java -jar` 报 `UnsupportedClassVersionError: class file version 61.0 vs 52.0`
- **根因**：系统默认 `java` 命令是 JDK 8（PATH 里 Oracle Java8 在前），但项目用 JDK 17
- **修复**：所有启动命令用 `"$JAVA_HOME/bin/java"`，不要直接用 `java`
- **JAVA_HOME**：`D:\javaEE\soft\jdk-17.0.9`

### 1.2 Maven resource filter + 中文注释
- **现象**：mvn package 失败，application.yml 有中文时 Maven 报 XML 解析错误
- **根因**：项目 pom 开了 resource filter，中文注释 `数据源` 在 Windows GBK 控制台混码
- **修复**：application.yml 用纯英文注释（`# datasource config`，不用 `# 数据源`）
- **备用**：url 里的 `&` 改成 `&amp;` 也可以（但英文化更彻底）

### 1.3 正确的 jar 执行命令
- **现象**：`java -jar training-api.jar` 报 "no main manifest"
- **根因**：pom 里用了 `<classifier>exec</classifier>`，真实可执行 jar 是 `*-exec.jar`
- **修复**：
  ```bash
  # 不要用
  java -jar training-admin/target/training-admin.jar  # ❌ 没有 manifest
  
  # 要用
  java -jar training-admin/target/training-admin-exec.jar  # ✅
  java -jar training-api/target/training-api-exec.jar      # ✅
  ```

### 1.4 GBK/UTF-8 编码问题
- **现象**：curl 发中文 JSON 时后端报 `Invalid UTF-8 middle byte 0xe7`
- **根因**：Windows Git Bash 控制台默认 GBK，curl 发送时编码错乱
- **修复**：发中文 JSON 时用 Python urllib：
  ```python
  import urllib.request, json
  data = json.dumps({"question":"考试怎么办"}, ensure_ascii=False).encode('utf-8')
  req = urllib.request.Request(url, data=data, headers={"Content-Type":"application/json"})
  resp = json.load(urllib.request.urlopen(req))
  ```
- **快速方案**：curl 参数全用 ASCII（写测试脚本时避开中文）

### 1.5 端口占用
- **现象**：`PortInUseException: Port 8080 already in use`
- **根因**：pkill 杀进程后 TIME_WAIT 状态未释放
- **修复**：`cmd //c "taskkill /F /IM java.exe"` 彻底杀

### 1.6 前端 vite 端口从 5173 改成 5174
- frontend-lead 静默改了 `vite.config.ts` 里的端口（避开占用）
- **影响**：浏览器访问前端用 `http://localhost:5174`

## 二、子代理（sub-agent）可靠性问题

### 2.1 子代理零文件崩溃
- **现象**：多次 `Agent` 工具派发 frontend-lead / backend-lead，子代理报告"完成"但磁盘上**零文件**
- **频率**：frontend 代理 2 次，backend-lead B6 1 次（`*.output` 文件 0 字节）
- **影响**：浪费大量时间调试 task-notification 假阳性
- **应对**：
  - **主线程直接写关键代码**（跟 frontend agent 一样，发现零文件后我自己建目录 + 写所有文件）
  - 子代理只做机械任务（改文案、格式整理）
  - 未来考虑用 `claude --dangerously-skip-permissions -p "..."` 替代（见下面 2.4）

### 2.2 任务取消的应对
- **现象**：用户偶尔中断当前 turn（"Interrupt"）
- **损失**：子代理继续跑完，但后续的汇报/修复代码没有执行
- **应对**：中断后每次检查一遍 `.output` 文件大小，0 字节就重新派发或主线程接管

### 2.3 学到的 agent 派发规范
- agent prompt **必须包含**：具体步骤、自检命令、验收标准
- agent prompt **必须说清楚**：output 路径、不要用中文注释、mvn install 后再启动
- 不要用 `run_in_background: false`（同步等待），用 background + task-notification
- B6 的 agent 最终其实写出了文件，只是日志 0 字节（mvn install 之前就退出了），经验是**只要 API 在磁盘上且 mvn install 通过**，即便子代理报失败也当成功

### 2.4 未来推荐：用 `claude -p` CLI 替代 Agent 工具
```bash
cd D:\A-Users\Desktop\zhongheshixun
claude -p "（具体的 prompt）" --dangerously-skip-permissions --model sonnet
```
这样：
- 不占用主线程 context
- 不会遇到"子代理零文件"问题
- 输出直接写入磁盘，可以用 worktree 隔离

### 3.2 分页 total 不设置

### 3.4 train-plan/page 返回 500（无 SQL 日志 + 无 Exception 日志）⭐ 新增

**现象**：
- `GET /admin/train-plan/page` 返回 `500 {"message":"系统繁忙，请稍后重试"}`
- `GET /admin/course/page` 正常返回 200
- Console/日志中**无 MyBatis SQL、无 Exception**（`log-impl: StdOutImpl` 已开，却什么都没打）

**调用链**：
```
JwtAuthenticationFilter.doFilterInternal
  → RbacUserDetailsService.loadUserByUsername
  → @PreAuthorize("hasAuthority('plan:read')")   ← TrainPlanController:35
  → TrainPlanService.page → TrainPlanMapper.selectTrainPlanPage
```

**我走过的弯路（已铁证排除）**：
| 序号 | 嫌疑 | 证据 | 结论 |
|------|------|------|------|
| ① | 缺 RBAC 表（sys_role/sys_permission） | `DESCRIBE sys_user` / `information_schema.tables` → 三表都在、列都在、admin.role_id=1 | ❌ 排除 |
| ② | 缺 `sys_user.role_id` 列 | `SHOW COLUMNS FROM sys_user` → role_id 存在 | ❌ 排除 |
| ③ | `sys_role_permission` 字段名错 | 实际是 `permission_id`（与 mapper.xml 一致） | ❌ 排除（我查询时自己写错列名导致误报） |
| ④ | JwtUtils HMAC 密钥异常 / IPage 序列化问题 | course:page 同样写法(`IPage<X>` + `@PreAuthorize`)却正常 | ❌ 排除 |

**真正根因（铁证）**：

MySQL `sys_permission` 表**没有 `plan` 模块的权限记录**：
```sql
SELECT module, COUNT(*) cnt FROM sys_permission GROUP BY module;
-- 返回 8 行：chapter/consult/course/exam/knowledge/question/stats/user
-- ❌ 没有 plan
```
`TrainPlanController` 上标了 `@PreAuthorize("hasAuthority('plan:read')")`，但 DB 中不存在 `plan:read` 这条权限 → 鉴权失败。

**为什么无 SQL 无 Exception**：
- 鉴权失败发生在 `ExceptionTranslationFilter` → `CustomAccessDeniedHandler` 这条链路
- 旧的 `JwtAuthenticationFilter` 里的 `catch(Exception e)` 只 `log.warn(e.getMessage())`（**丢弃堆栈**）后还 `chain.doFilter(...)`，把异常吞成了 500
- SQL 也未执行（请求还没到 Mapper 就被拦下）

**修复**：

**(a) 补 DB 权限**（两条 INSERT + 一条关联，jar 不重启即生效）：
```sql
USE training;
INSERT IGNORE INTO sys_permission (perm_code, perm_name, description, module) VALUES
  ('plan:read',  '培训计划查看', '培训计划列表/详情',   'plan'),
  ('plan:write', '培训计划编辑', '计划增删改+关联课程', 'plan');

INSERT IGNORE INTO sys_role_permission (role_id, permission_id)
SELECT 1, id FROM sys_permission WHERE perm_code IN ('plan:read','plan:write');
```

**(b) 强化 filter**（`training-admin/.../security/JwtAuthenticationFilter.java:102-107`）：
```java
// 旧（错误：吞异常+丢堆栈）
} catch (Exception e) {
    log.warn("[RBAC] 认证流程异常：{}", e.getMessage());
    SecurityContextHolder.clearContext();
    chain.doFilter(request, response);
}

// 新（暴露真因）
} catch (Exception e) {
    log.error("[RBAC] 认证流程异常，URI={}", request.getRequestURI(), e);
    SecurityContextHolder.clearContext();
    response.sendError(HttpServletResponse.SC_UNAUTHORIZED, e.getMessage());
    return;
}
```

**(c) 防御性改造**（避免同类 Jackson 序列化坑）：
- `TrainPlanController.page`：`Result<IPage<TrainPlan>>` → `Result<Page<TrainPlan>>`（具体类，非接口）
- 同步改 `TrainPlanService` 接口和 `TrainPlanServiceImpl` 返回类型

**(d) 同步** `docs/database.sql`：补建 3 张 RBAC 表 + 15 条权限 DDL + admin 的 `role_id=1` UPDATE，下次重建库不再踩坑

**教训（「500 无日志+同类接口正常」排查范式）**：
> **同类接口的差异点就是根因**。按优先级：
> 1. **注解 authority 不同**（本案例：`plan:read` vs `course:read` → DB 压差查询）
> 2. 参数类不同（`TrainPlanPageQuery` vs `CoursePageQuery`）
> 3. 服务链不同
>
> 遇到「无日志、无 SQL」，第一件事不是改代码——而是**强化 filter 让异常现出真身**（log.error 打堆栈 + sendError 中断）。否则永远在猜。

**修复后状态**：`GET /admin/train-plan/page` 返回 200 + 能看见培训计划数据 ✓

## 八、2026-07-09 新增 case

### 8.2 /admin/teacher/page 返回 500（与 train-plan 同类坑，RBAC 权限缺失）

**现象**：
- `GET /admin/teacher/page` 返回 `500 {"message":"系统繁忙，请稍后重试"}`
- 同类 list（course/page、exam/page）正常

**根因（铁证）**：
- `TeacherController` 标了 `@PreAuthorize("hasAuthority('teacher:read')")`
- `SELECT module, COUNT(*) cnt FROM sys_permission GROUP BY module` → 8 个模块全无 `teacher`
- admin 拿不到 `teacher:read` → 鉴权失败 → 被 filter 吞成 500

**修复**（jar 不重启即生效，与 train-plan 同范式）：
```sql
USE training;
INSERT IGNORE INTO sys_permission (id, perm_code, perm_name, description, module) VALUES
  (19, 'teacher:read',  '讲师查看', '讲师列表/详情可见',   'teacher'),
  (20, 'teacher:write', '讲师编辑', '新增/修改/删除讲师',   'teacher');

INSERT IGNORE INTO sys_role_permission (role_id, permission_id)
SELECT 1, id FROM sys_permission WHERE perm_code IN ('teacher:read','teacher:write');
```

**防回归**：同步 `training-parent/.../db/database.sql` 与 `docs/database.sql`，在种子数据的 sys_permission 里补 `teacher:read`/`teacher:write`（id 16/17），并把「ADMIN 绑定全部 N 条权限」从 15 改成 17。下次重建库不再踩。

**修复后**：`/admin/teacher/page?pageNum=1&pageSize=5` 返回 200 + 显示张教授、李主任数据 ✓

**扩展教训**：
> 每新增一个 `@PreAuthorize("hasAuthority('xxx:read')")` 控制器，**必须在 sys_permission 种子数据里同步补该模块权限**，否则 list 接口必然 500。建议建表后在 database.sql 里维护完整 perm_code 清单，新增模块时一次性补全。

### 9.2 学到的范式（3 条）

**(a) 非持久化字段模式**：当 schema 演进比前端需求慢时，用 `@TableField(exist = false)` 字段 + Service/Controller 注入默认值，可避免污染数据库且前端拿到一致字段。后续 schema 升级时只需删除 `exist=false` 即可平滑迁移。

**(b) setInterval 计时陷阱**：setInterval 在浏览器标签页切走后会被节流（最低 1Hz），导致"每秒 -1"在恢复时跳过若干秒，剩余时间虚高。**正确模式**：记录 `endTimestamp`，setInterval 仅做触发刷新，`remaining = (endTimestamp - Date.now()) / 1000`。切回标签页时下一 tick 自动按真实时间差补正。

**(c) "字段重名不同值"是隐性 bug 源**：`exam.status` 在管理端（DB 列：0 草稿/1 已发布/2 已下架）和学员端（`ExamListVO.status`：0 未开始/1 已完成/2 已批阅）含义完全不同，但都叫 `status`。后端通过独立 VO 类做字段隔离，前端不应把两个 `status` 当作同一个字段做比较。

### 9.3 踩过的构建坑

- `vue-tsc --noEmit` 严格类型检查会暴露原代码隐藏 bug。例如 `formData` 定义缺 `examType` 字段（模板 v-model 但 data 无），构建时 TS 报错，**必须补字段才能通过**。

#### 11.1.2 教师权限对齐教学完整版（SecurityConfig 8 模块通配符）

**现象**：原 SecurityConfig 逐个端点写法（`/admin/course`、`/admin/course/page`、`/admin/course/{id}` 三行）覆盖不全，教师登录 5176 后部分教学功能被 403。

**修复**（[SecurityConfig.java](file:///D:/A-Users/Desktop/zhongheshixun/training-parent/training-admin/src/main/java/com/training/admin/config/SecurityConfig.java)）：
```java
// 教学完整版：TEACHER 可访问 8 个教学模块（/** 通配符）
.antMatchers("/admin/course/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/chapter/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/knowledge/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/question/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/exam/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/plan/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/consult/**").hasAnyRole("ADMIN", "TEACHER")
.antMatchers("/admin/stats/**").hasAnyRole("ADMIN", "TEACHER")
// 用户管理、讲师管理仅 ADMIN
.antMatchers("/admin/**").hasRole("ADMIN")
```
同步 `database.sql` 补授 consult:read/write 给 TEACHER（TEACHER 15 条 → 17 条，含全量）。

### 11.5 学到的范式（2 条）

**(a) SecurityConfig 端点覆盖要彻底**：用 `/admin/course/page`、`/admin/course/{id}` 逐个列端点，漏一个就 403。**正确模式**：用 `/admin/course/**` 通配符一行覆盖整模块，新增子端点零维护。M12 阶段用 `hasAnyRole("ADMIN","TEACHER")` 三行写法已踩坑，本次升级为 `/**` 通配符彻底解决。
