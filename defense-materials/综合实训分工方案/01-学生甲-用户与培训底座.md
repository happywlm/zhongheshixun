# 学生甲 — 用户体系 + 培训底座（系统入口 + 治理）

> **综合实训角色**：学生甲  
> **业务域**：用户体系 + 培训底座  
> **核心叙事**：「系统怎么认人 → 人分哪几种 → 培训怎么组织」

---

## 一、业务定位

你是整个系统的**入口和底座**。你的模块回答三个问题：

1. **谁进来了？** — 登录认证（JWT 签发与校验）
2. **他是谁？** — 用户角色（管理员 / 讲师 / 学员 3 种 RBAC）
3. **培训怎么组织的？** — 培训计划编排（训练计划 → 关联课程 → 流程控制）

其他 3 位学生（乙/丙/丁）的前置依赖都是你：
- 乙的课程需要"学员"角色才能报名
- 丙的考试需要"学员"角色才能答题
- 丁的咨询需要"学员"角色才能提问

## 五、综合实训深度点（3 个）

### 5.1 JWT 认证全流程（核心亮点，必须能画时序图）

**代码位置**：`JwtUtils.java`（training-common 公共模块） + `AuthController.java` + `JwtAuthenticationFilter.java` + `RbacUserDetailsService.java`

> JWT 工具类放在 `training-common` 而非 `training-admin`，是为了让 `training-api`（9899）也能共享同一套签名/验签逻辑，保证跨服务 token 互通。

**关键代码**：

```java
// JwtUtils.java - 生成 token（HMAC-SHA256）
public String generate(Long userId, String username, String role, Long roleId) {
    Map<String, Object> claims = new HashMap<>();
    claims.put("userId", userId);
    claims.put("username", username);
    claims.put("role", role);              // 旧字段，向前兼容
    if (roleId != null) {
        claims.put("roleId", roleId);      // RBAC 主路径
    }
    return Jwts.builder()
            .setClaims(claims)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + expire)) // 2h
            .signWith(getKey(), SignatureAlgorithm.HS256)                // Key 由 secret 字符串派生
            .compact();
}

// 密钥派生（不是直接用字符串签名）
private SecretKey getKey() {
    return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
}

// JwtUtils.java - 解析 token
public Claims parse(String token) {
    return Jwts.parserBuilder()
            .setSigningKey(getKey())
            .build()
            .parseClaimsJws(token)
            .getBody();
}

// JwtUtils.java - 验 token
public boolean validate(String token) {
    try {
        parse(token);  // 验签 + 验过期，失败抛 JwtException
        return true;
    } catch (JwtException | IllegalArgumentException e) {
        return false;
    }
}

// JwtAuthenticationFilter.java - 注入 SecurityContext
// token 解析后，调用 RbacUserDetailsService 加载 LoginUser
// LoginUser 持有：roleCode（如 ADMIN）+ 权限码列表（如 user:read, plan:write）
// authorities 中同时包含 ROLE_ADMIN（角色）和 user:read（权限码）
```

**教学点**：
- 为什么 JWT 比 Session？（无状态、分布式友好，9898/9899 共享同一 secret 即可互验）
- secretKey 配在哪里？（`application.yml` → `training.jwt.secret`，9898/9899 必须一致）
- token 过期怎么办？（`parse` 抛 `ExpiredJwtException`，过滤器返回 401，前端跳登录）
- token 被盗怎么办？（HTTPS + 短有效期 2h + 后续可扩展 refresh token）
- 为什么用 `parserBuilder()` 而不是 `parser()`？（jjwt 0.11.x 新 API，旧 API 已弃用）

### 5.2 RBAC 4 表权限模型（核心亮点）

**角色定义**（sys_role 表）：

| role_code | role_name | 权限范围 |
|-----------|-----------|----------|
| **ADMIN** | 系统管理员 | 全部功能（user/teacher/plan/course/exam/consult 所有读写权限） |
| **TEACHER** | 讲师 | 课程管理 + 试题管理 + 查看学员进度 + 答疑 |
| **STUDENT** | 学员 | 课程学习 + 考试 + 个人中心 + 咨询提问 |

**权限控制实现**（4 表 + Spring Security 双层鉴权）：

```java
// Controller 层 - 细粒度权限码（推荐，命中 sys_role_permission 关联表）
@PreAuthorize("hasAuthority('user:read')")
@GetMapping("/page")
public Result<IPage<SysUser>> page(UserPageQuery query) { ... }

@PreAuthorize("hasAuthority('plan:write')")
@PostMapping
public Result<Boolean> create(@RequestBody @Valid TrainPlanForm form) { ... }

// 角色级粗粒度（兜底，URL 级别配置在 SecurityConfig）
// /admin/** 要求 ROLE_ADMIN
// 教学相关路径 hasAnyRole("ADMIN","TEACHER")
```

**权限加载链路**：

```
请求带 token
   ↓
JwtAuthenticationFilter 解析 token → 拿到 username
   ↓
RbacUserDetailsService.loadUserByUsername(username)
   ↓
1. sysUserMapper.selectByUsername → 拿到 user.roleId
2. sysRoleMapper.selectById(roleId) → 拿到 roleCode（如 ADMIN）
3. sysPermissionMapper.selectByRoleId(roleId) → 拿到权限码列表 [user:read, user:write, plan:read, ...]
   ↓
组装 LoginUser：
   authorities = [ROLE_ADMIN, user:read, user:write, plan:read, plan:write, ...]
   ↓
注入 SecurityContext
   ↓
@PreAuthorize("hasAuthority('user:read')") 命中 authorities 中的 user:read → 放行
```

**教学点**：
- 为什么用 4 表 RBAC 而不是硬编码角色字符串？（权限可配置，新增角色只改数据库不改代码）
- `hasAuthority('user:read')` 和 `hasRole('ADMIN')` 区别？（前者查权限码，后者查 `ROLE_` 前缀角色，二者在 LoginUser 中都存在）
- 怎么防越权？（接口层 `@PreAuthorize` 是最后一道防线，不能只靠前端隐藏按钮）
- 为什么 `sys_user.role_id` 是单角色设计？（本期作业简化，一人一角色；如需多角色可加 `sys_user_role` 中间表）

### 5.3 3 种培训模式 + 3 类考试

**培训模式**（docx 3.1 强制要求）：

| 模式 | 实现方式 | docx 原文 |
|------|----------|-----------|
| 公开课 | `course_type = 1` | 允许学员自主选择学习 |
| 必修课 | `course_type = 2` | 由管理人员分配，针对不同组织机构提供个性化、定制化课程 |
| 计划流程性培训 | `train_plan` + `plan_course.sort_order` | 课程和测评通过一定顺序关系和条件关系组合为完整的学习培训方案 |

**考试类型**（docx 3.2 强制要求）：

| exam_type | 考试类型 | docx 原文说明 |
|-----------|---------|-------------|
| 1 | 课程考试 | 课程关联，学完课程后参加，未通过可重考 |
| 2 | 计划内考试 | 作为计划里面的子活动，可作为流程通过条件或计划完成条件 |
| 3 | 单独考试 | 独立于课程/计划，一般可用于定期测评 |

## 九、常见 Bug 与解决

| 现象 | 原因 | 解决 |
|------|------|------|
| 登录失败 "密码错误" | 密码是 `123456` 不是 `password` | 改密码或改 DB hash |
| token 过期 401 | 2h 无操作 | 重新登录 |
| 按钮能看到但点完 403 | 前端菜单显示，后端 role 校验失败 | 接口层检查 role 是否匹配 |
| 添加用户失败 "用户名已存在" | username 唯一约束 | 换 username |
| 删除用户后还能登录 | 逻辑删真删？ | 确认是 `deleted=1` 不是真删 |
| 培训计划关联课程失败 | course_id 不存在 | 先建课程再关联 |
